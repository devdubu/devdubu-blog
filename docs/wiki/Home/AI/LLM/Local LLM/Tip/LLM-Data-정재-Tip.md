결론부터 말씀드리면, 이 문제의 원인은 RAG 시스템의 **'검색(Retrieval)' 단계가 아니라, LLM이 검색된 정보를 바탕으로 답변을 '생성(Generation)'하는 단계**에 있습니다. 더 구체적으로는, **모델이 복잡한 마크다운 테이블(Table) 형식을 잘 해석하지 못하기 때문**입니다.

###  ✅ 근본적인 원인과 해결책

#### 원인: 마크다운 테이블의 구조적 복잡성

`rag_core.py`의 청크 분할(`split('\n## ')`) 로직은 '주요 속성' 섹션을 하나의 덩어리로 잘 찾아옵니다. 하지만 LLM은 `|`, `-` 등의 문자로 이루어진 테이블의 **구조를 이해하고 정보를 추출하는 능력이 상대적으로 떨어집니다.** 코드 블록처럼 명확한 시작과 끝이 있는 정형화된 데이터는 잘 처리하지만, 테이블은 그렇지 않습니다.

#### 해결책: LLM이 이해하기 쉬운 형태로 원본 문서 수정

가장 확실하고 좋은 해결책은 원본 마크다운 문서(`prompt` 폴더 안의 파일)의 형식을 테이블이 아닌, **일반 텍스트나 리스트 형태**로 바꾸는 것입니다.

- **수정 전 (테이블 형식)**
```Markdown
| 속성 | 설명 | 타입 | 기본값 |
| :--- | :--- | :--- | :--- |
| `v-model` | 선택된 항목의 값과 바인딩됩니다. | `string` | `null` |
```
- **수정 후 (리스트 형식 - 추천)**
```Markdown
## 주요 속성 (Props)
    
- **v-model**:
	- 설명: 선택된 항목의 값과 바인딩됩니다.
    - 타입: `string` | `null`
	- 기본값: `null`
- **data**:
	- 설명: 셀렉트 박스의 옵션 목록을 구성할 객체 배열입니다.
    - 타입: `Array<Object>`
    - 기본값: `[]`    
... (이하 동일) ...
```
이렇게 원본 문서의 형식을 바꿔주면, 비서(LLM)가 훨씬 이해하기 쉬운 자료를 받게 되어 '속성 정보'에 대해서도 명확하게 요약하고 답변해 줄 수 있게 됩니다.
