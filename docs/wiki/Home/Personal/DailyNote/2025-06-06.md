# Backend 현 상황

## 조회
조회 자체는 auth를 보유하고 잇다면 가장 최하위 권한이기 때문에, permission에 대한 체크 보단 존재 유무 에 대한 체크만 해도 무방하다

- [x] 메뉴 필터 로직
- [ ] 프로젝트 필터 로직
	- 메뉴 필터 로직과 유사하게 진행
	- 대신 menuCd 값이 `none` 값일 때만 체크
- [ ] 조직 필터 로직
	- 프로젝트 필터 로직과 유사하게 진행
	- projCd 값이 `none` 인 값만 비교
- [ ] 사용자 필터 로직 - Team에 대한 개념은 차차 고민해보기로
	- 최종으로 가지고 있는 권한에 따라서 조회하는 조건을 달리 진행함
		- 최종 권한은 `none` 값을 기준으로 매김
	- 최종 권한이 project 일 경우 Project에 한정해서 사용자를 추출함
	- 최종 권한이 organization 일 경우는 organization에 사용자를 추출함
	- 사용자는 생성하게 된다면 auth를 통해서 받은 cd 값을 기준으로 organization과 project를 배치
		- 만약 이 값이 없다면 core, core project에 기본적으로 배치
- [ ] 역할 필터 로직
	- permission 중에서 1111 이 있는 최종 권한이 projCd 값만을 보여주며
	- projCd보다 낮은 수준의 권한만 필터링 해서 보여주기
	- 마찬가지로 조직도 위 로직을 따라간다.
- [ ] 권한 필터 로직
	- 역할 필터 로직과 동일하게 진행


## 수정 및 삭제 권한 부여

- [ ] 메뉴 로직
	- auth를 통해서 projCd를 받는다.
	- 해당 사용자가 projCd중에 permission이 1100, 1110 이상일 경우 저장 or 삭제
- [ ] 조직 로직
	- 조직에 대한 수정 및 삭제는 auth로 받은 OrgCd값과 수정 요청을 진행한 orgCd를 대조함
	- Permission이 1100 이상일 경우 저장 가능
	- 삭제 부터는 관리자 4레벨 이상 부터 가능
- [ ] 프로젝트 로직
	- auth를 통해서 projCd를 받는다.
	- 해당 사용자가 projCd중에 permission이 1100 이상일 경우 수정
	- 삭제는 orgCd 레벨 1110이상인 경우 가능
- [ ] 사용자 로직
	- 변경하려는 데이터가 속한 proj or org 에 따라서 권한 체크 후 저장하기
