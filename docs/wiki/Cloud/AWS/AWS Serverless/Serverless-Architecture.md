# Serverless Architeture란?
![Pasted-image-20230810123624.png](/img/이미지 창고/Pasted-image-20230810123624.png)
- 현 내용에 대해서는 기본적으로 서버리스에 대해서 다룰 예정입니다.
- 서버리스는 서버가 없다는 것이 아닌 <mark>관리할 서버가 없다</mark> 라는 것에 가깝습니다.
- 서버를 설정하고, 서버를 구성하는 것들에 대한 공수가 줄어드는 만큼, Cloud Native한 구성이 가능합니다.
## 기존의 기술
![Pasted-image-20230810124226.png](/img/이미지 창고/Pasted-image-20230810124226.png)
### 자체적 시스템 설계
- 전산실 혹은 인프라 측에서 시스템 자체를 전반적으로 모두 관리하는 것을 의미한다.
- 공간, 하드웨어, 네트워크, 운영체 등등 모두 관리를 해주는 것을 자체적 시스템 설계라고 합니다.
- 시스템이 커지게 된다면, 관리자, 즉, 인력에 대한 돈이 나간다는 것입니다.

### IaaS (Infrastructure as a Service)
- 네트워크, 공간, 하드웨어 등등 모든 것들을 직접 설치할 필요가 없습니다.
- 인프라르 가상화하여, 직접 네트워크, 운영체제 등을 쉽게 셋팅이 가능해집니다.
- 관리자 패널에서, 사용자는 가상화 시스템을 활용해서 시스템을 셋팅할 수 있게 됩니다.

### PaaS (Platform as a Service)
- IaaS 에서 한번 더 추상화 된 개념입니다.
- 네트워크, 런타임까지 제공해주고, 애플리케이션만 배포하게 된다면 바로 배포가 가능합니다.
	- 대표적으로 Elastic Beanstalk, Azure Appservice 등이 있습니다.
	- 이를 활용하면 Auto scaling과 같은 scale out 프로그램을 쉽게 사용 가능합니다.



:::note 서버리스 아키텍쳐?
- 애플리케이션이 필요한 경우에만 시작
- 이벤트가 구동을 위한 애플리케이션 코드를 트리거 하변, 퍼블릭 클라우드 공급업체가 신속히 해당 코드에 대한 리소스를 할당
- 코드 실행이 종료되면 비용도 청구 되지 않음
- 운영 체제 및 파일 시스템 관리, 보안 패치, 부하 분산, 용량 관리, 스케일링, 로깅, 모니터링과 같은 일상적인 태스크를 모두 클라우드 서비스 제공업체에 이관
- 완전한 서버리스 애플리케이션은 물론, 일부 서버리스로 일부는 전통적인 마이크로 서비스 구성 요소로 이루어진 애플리케이션 구축
 

:::

# BaaS(Backend as a Service)
:::tip BaaS?
- 애플리케이션 개발에 있어서 필요한 다양한 기능들을 API로 제공해 줌으로서, 필요한 기능을 쉽고 빠르게 구현 할 수 있게 함

:::

## PaaS 와 주요 차이점
- 개발자의 생산성을 높이고 <mark>운영 비용</mark>을 줄 일 수 있다.
- 인프라를 명시적으로 설명할 필요를 줄여줌으로써 DevOps 도입을 지원
- 3rd party BaaS 오퍼링의 모든 구성 요소를 통합해 애플리케이션 <mark>개발을 간소화</mark>
- 컴퓨팅 시간에 대한 비용을 지불하기 때문에 운영 비용이 낮아짐

:::success 장점
- 개발 속도가 매우 빠름
- 개발 비용이 매우 저렴함
- 서버가 필요 없고, 기반 시설을 관리할 필요가 없음

:::

:::error 단점
- 커스텀 코딩에 비교하여 낮은 유연성
- 커스텀 백엔드와 비교하여 낮은 레벨의 맞춤화
- 비 개방형 소스 플랫폼을 위한 벤더 종속

:::

# FaaS(Function as a Service)
:::tip FaaS?
- 프로젝트를 여러 개의 함수로 쪼개서, 매우 거대하고 분산된 컴퓨팅 자원에 여러분이 준비해둔 함수를 등록하고
- 이 함수들이 실행되는 횟수 만큼 비용을 내는 방식

:::

## PaaS와 주요 차이점
- 서버 시스템에 대해서 신경 쓰지 않아도 된다는 점이 PaaS와 유사
- <mark>PaaS</mark>의 경우엔, 전체 애플리케이션을 배포하며, 일단 어떠한 서버에서 당신의 애플리케이션이 24시간 동안 계속 돌아가고 있다.
- 반면, <mark>FasS</mark>는, 애플리케이션이 아닌 함수를 배포하며, 계속 실행되고 있는 것이 아니라 특정 이벤트가 발생했을 때 실행되며, 실행이 되었다가 작업을 마치면 종료된다.

:::success 장점
- **비용**
- 필요할 때만 함수가 호출되어 처리되며 비용이 절약됨
- **인프라 관리**
- 네트워크, 장비 등에 대한 구성 작업을 신경 쓰지 않아도 됨
- **인프라 보안**
- OS 업데이트, Meltdown 취약점 보안 패치 등을 신경 쓰지 않아도 됨
- **확장성**
- 특정 조건에 따라 자동으로 확장되는 것이 아니라 자동 확장

:::

:::error 단점
- **제한**
- 함수에서 사용할 수 있는 자원에 제한이 있다.
- **FaaS 제공 사의 강한 의존**
- AWS, Azure, Google 등 FaaS 제공사에 강한 의존


:::
